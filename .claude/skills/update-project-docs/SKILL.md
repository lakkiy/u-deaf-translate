# 更新项目文档

把刚刚发生的决策、踩坑、修复同步到项目文档里，避免知识只留在对话历史中。

## 何时调用（主动调用）

出现以下情况之一就该调，**不要等用户提醒**：

- 做了一个非显然的设计决策（"为什么不用 X 而是 Y"）
- 修了一个 surprising 的 bug（"以为是 A，实际是 B"）
- 加了一个 workaround 应对外部限制（浏览器/API/库的怪行为）
- 改了工具链（构建工具、依赖、配置）
- 学到一个会影响未来决策的事实

**不要调用：**
- 简单 bug 修复（typo、明显的逻辑错）
- 例行代码改动（重构、命名）
- 已经写在文档里的内容
- 一两轮对话内还不确定的设计——等用户确认后再记

## 调用流程

1. 读 `docs/decisions.md` 和 `docs/pitfalls.md`，了解现有内容
2. 找出本次会话里值得记录的新条目
3. 决定归到 `decisions.md`（"为什么这么做"）还是 `pitfalls.md`（"曾经踩过的坑"）
4. 选合适的 section（架构 / 性能 / 样式 / 工具链 / 范围 等），用一致的格式追加
5. 不要重复已有内容；如果是更新已有条目，直接 Edit 不要追加新条目
6. 不要 commit，用户会自己 commit

## 格式约定

### `decisions.md` 格式
```markdown
### 决策标题（一个动词短语或名词短语）
一段说明（1-3 句），核心是"为什么这么做"和"考虑过的替代"。引用具体行为或代码，不空谈。
```

### `pitfalls.md` 格式
```markdown
### 坑的一句话标题
- **现象：** 看到的症状（用户视角）
- **真相：** 实际原因（技术层面）
- **修复：** 怎么解决的（具体改动，引用文件/函数）
```

### Sections
归类参考已有 sections（不必死守，但保持类似分类）：
- `decisions.md`: 架构 / 性能 / 样式 / 工具链 / 范围 / 错误处理 / 隐私
- `pitfalls.md`: Translator API / DOM-CSS / TypeScript-构建 / 体验细节 / AI 模型行为 / 浏览器限制

## 写作风格

- 中文为主
- 每条尽量短——一段话或三个 bullet，最多 3-5 句
- 重在"为什么"和"非显然之处"，不复述代码做了什么
- 不要"改进"、"优化"这类空话，描述具体改动
- 引用代码用反引号；引用文件用 `path:line` 格式

## 修改后

在响应里告诉用户加了什么新条目（不要复述全部内容，只说新增/更新了什么 + 一句话总结）。
